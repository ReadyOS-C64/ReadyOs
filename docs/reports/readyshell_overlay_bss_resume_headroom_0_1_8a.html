Failure mode: the pre-fix overlay linker config placed BSS at __ONCE_RUN__. That made resident BSS cover $9ACA-$9B01, the cc65 ONCE startup block. First disk launch could still appear to work because the PRG load supplied fresh startup bytes. After ReadyOS saved the app window to REU, however, the snapshot contained BSS-clobbered startup bytes. Warm reentry restored that corrupted window and jumped back through cc65 startup before main().
Fix: start resident BSS at __ONCE_RUN__ + __ONCE_SIZE__ and bound the resident BSS memory area below __OVERLAY_LOADADDR__. That preserves cc65 startup across REU save/restore and keeps the existing fast warm-resume path; it does not force a disk reload.
Immediate delta, not 0.1.5: this report compares the unsafe Readyshell overlay config immediately before the last change with the fixed 0.1.8A map. The resident BSS growth window lost exactly 56 bytes, the ONCE block that had previously been double-booked as BSS. The true tightest ReadyShell headroom remains the overlay payload limit: OVERLAY1 has $0387 / 903 bytes before $C600.

ReadyShell is not a normal cfg/ready_app.cfg application. It uses cfg/ready_app_overlay.cfg, releases overlays into sidecar PRGs, and reserves REU banks $40-$43 for overlay and debug state. It also relocates the cc65 C stack separately at runtime, so the old __STACKSIZE__ comment was not the real resident BSS safety model. The real resident boundary is the overlay load-address window at $A1BE.

Resume Path Evidence

Failure before the fix

  • First disk launch entered ReadyShell normally because the PRG load supplied fresh cc65 startup bytes.
  • Runtime BSS then reused $9ACA-$9B01, overwriting the resident ONCE startup block.
  • Returning to launcher saved the active app window $1000-$C5FF to the app REU bank, including the corrupted ONCE range.
  • Warm reentry restored the corrupted snapshot and jumped to $1000.
  • The failure looked like an immediate return to BASIC on second entry, matching a startup failure before useful app code could run.

Proof after the fix

  • Rebuilt all release variants through bash ./run.sh --build-all, producing checked-in 0.1.8A artifacts.
  • Launched the full precog-dual-d71 ReadyOS profile with bash ./run.sh --profile precog-dual-d71 --skipbuild.
  • ReadyShell was tested through the normal ReadyOS launcher and quick app changing flow, not by direct app launch.
  • User acceptance result: ReadyShell reentry and shell usage were reported as "working perfect".
  • No disk reload requirement was introduced; the fix preserves the warm REU snapshot path by making the snapshot valid.
Acceptance result: this honors the ReadyOS quick app changing requirement. The app resumes from the saved REU image; the linker fix prevents the saved image from containing clobbered cc65 startup bytes.

ReadyShell stack nuance: the old config comment said it kept the C stack below the overlay window. In practice ReadyShell calls its runtime stack relocation separately, while the resident linker boundary needed to protect ONCE and the overlay load-address bytes. The report therefore separates resident BSS headroom, overlay payload fit, and raw app-window gap. For practical ReadyShell growth, the overlay payload limit is currently tighter than resident BSS.
